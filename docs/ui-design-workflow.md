# UI 设计阶段标准化流程

## 概述

本文档定义从需求分析到开发交接的标准流程，确保 UI 设计阶段完整、可控、可交接。

---

## 整体流程

```
┌─────────────────────────────────────────────────────────────────┐
│                      UI 设计阶段                                 │
│  需求分析 → 信息架构 → 流程设计 → 线框图 → 高保真 → 开发交接      │
└─────────────────────────────────────────────────────────────────┘
                              ↓
                    ✅ 满足退出条件
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│                      开发阶段                                    │
└─────────────────────────────────────────────────────────────────┘
```

---

## 阶段一：需求分析与信息架构

### 目标

理解业务需求，确定功能模块和页面结构。

### 产出物

| 产出物 | 内容 | 格式 |
|-------|------|------|
| 功能模块清单 | 按优先级排序的模块列表（P0/P1/P2） | 表格 |
| 信息架构图 | 页面层级与导航结构 | 树状图/脑图 |
| 数据字典初稿 | 核心实体初步定义 | 表格 |

### 功能模块清单模板

| 模块编号 | 模块名称 | 功能描述 | 优先级 | 前后端分工 |
|---------|---------|---------|--------|-----------|
| M01 | 模块名 | 详细功能描述 | P0/P1/P2 | 前/后/全栈 |

### 退出条件

- [ ] 所有 P0 功能已识别并确认
- [ ] 页面数量已确定（不超过预估 20% 偏差）
- [ ] 产品负责人已确认信息架构

---

## 阶段二：流程设计与线框图

### 目标

明确用户流程，验证页面布局和跳转逻辑。

### 产出物

| 产出物 | 内容 | 格式 |
|-------|------|------|
| 核心业务流程图 | 从数据接入到查询的完整链路 | ASCII 图/Mermaid |
| 页面流程图 | 页面间跳转关系 | 流程图 |
| 线框图 | 每个页面的基本布局与元素 | 灰度线框 |

### 业务流程图示例

```
[数据源接入] → [本体建模] → [数据映射] → [版本发布]
     ↓              ↓             ↓            ↓
  元数据扫描     定义对象      表→对象      发布审批
     ↓              ↓             ↓            ↓
  连接测试       建立关系      映射验证      生成快照
```

### 退出条件

- [ ] 核心用户流程已走通（无死路/无逻辑断点）
- [ ] 页面数量与实际开发量匹配
- [ ] 产品负责人已确认所有线框图

---

## 阶段三：高保真原型

### 目标

交付视觉完整、可交互的原型，用于评审和开发参考。

### 产出物

| 产出物 | 内容 | 格式 |
|-------|------|------|
| 设计系统 | 色彩/字体/间距/组件规范 | CSS/Design Tokens |
| 高保真页面 | 所有页面的视觉效果 | HTML/CSS/设计稿 |
| 页面说明文档 | 每个页面的功能描述、前后端分工 | 表格 |

### 页面清单模板

| 页面名称 | 文件路径 | 功能描述 | 前端组件 | 后端接口 | 优先级 |
|---------|---------|---------|---------|---------|--------|
| 仪表盘 | index.html | 平台概览、统计、快捷入口 | Card, Table, Stats | /api/dashboard | P0 |

### 退出条件

- [ ] 所有页面已完成（含 P0 全部，P1/P2 可选）
- [ ] 路由逻辑已验证（无 broken links）
- [ ] 组件状态已覆盖（Normal/Hover/Active/Disabled/Loading/Error/Empty）
- [ ] 产品 + 开发 + 测试三方评审通过

---

## 阶段四：开发交接

### 必须交付物

| 交付物 | 检查项 |
|-------|-------|
| 设计系统文档 | 色彩、字体、间距、阴影规范 |
| 组件规格说明 | 每个组件的尺寸、状态、行为 |
| 交互规范 | 动画时长、缓动函数、过渡效果 |
| API 接口文档 | 请求/响应格式、错误码 |
| 数据模型定义 | 核心实体的字段、类型、关系 |
| 页面清单 | 14+ 页面的功能说明、前后端分工 |

## 各阶段时间参考（示例：两周迭代）

| 天数 | 阶段 |
|------|------|
| Day 1-2 | 需求分析 + 信息架构 |
| Day 3-5 | 流程设计 + 线框图 |
| Day 6-8 | 高保真原型 |
| Day 9-10 | 评审 + 交接文档 |
| Day 11-12 | 缓冲 + 返修 |
| Day 13-14 | 开发交接 + kickoff |

---

## 常见问题预防

| 问题 | 预防措施 |
|------|---------|
| 开发阶段频繁改设计 | 线框图阶段充分评审，高保真需三方签字 |
| 组件状态遗漏 | 组件规范阶段逐项检查清单 |
| API 格式不确定 | 开发前先输出 API 文档初稿，与后端对齐 |
| 页面数量失控 | 线框图阶段锁定页面范围，变更需走变更流程 |
| 路由逻辑错误 | 高保真阶段逐一验证所有跳转路径 |
